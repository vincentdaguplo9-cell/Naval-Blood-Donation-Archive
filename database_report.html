Speaker Scripts

Daguplo

“This defense focuses on our relational schema design. We normalized the data to avoid duplication, enforced referential integrity with foreign keys, and separated transactional history from master records for auditability.”

“We model donors, units, and transactions so every donation creates a unit and a transaction record, giving traceability from donor → unit → transaction.”

Lamoste

“Here is the ERD. The donor table is the master entity. blood_unit_table references donor_id, and donation_transaction_table references both donor_id and unit_id. This prevents orphan records and preserves history.”

“Constraints: NOT NULL for critical fields, UNIQUE on username, and FK constraints with RESTRICT to prevent deleting donors that have donation history.”

Ocampo

“Example SQL: INSERT into donor_table for new donors; INSERT into blood_unit_table for each donation; INSERT into donation_transaction_table for audit trail; UPDATE to change unit status in inventory.”

“We use prepared statements at the DAO layer to avoid SQL injection and to standardize data access patterns.”

Fuentes

“Reporting queries join donor_table, blood_unit_table, and donation_transaction_table to produce donor lists, inventory status, and history.”

“We can add indexes on donor_id and transaction_date to improve query performance for reports.”

Cator

“Admin security uses a password_hash + password_salt schema, plus failed_attempts and lock_until for lockout.”

“The database supports upgrade-on-login: legacy plaintext passwords are migrated to hashed values after the first successful login.”

Key Tables

  • donor_table — personal info and eligibility
  • blood_unit_table — volume, dates, status
  • donation_transaction_table — audit trail
  • admin_table — secure login
